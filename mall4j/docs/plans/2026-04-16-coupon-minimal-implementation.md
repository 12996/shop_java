# 优惠券最小功能实现计划

## 1. 背景

本计划基于已确认的设计文档：

- `docs/plans/2026-04-16-coupon-minimal-design.md`

目标是在当前项目中落地最小可用优惠券能力，只支持固定减金额优惠券，并保持实现范围可控、链路闭环完整。

## 2. 已确认约束

- 只做固定减金额优惠券
- 一张订单只允许使用一张优惠券
- 采用单表 `tz_coupon`
- 同时支持后台发券、用户领券
- 后台模块独立新建，但实现结构复用现有 `Controller + Service + Mapper + Model + DTO/VO` 模式
- 不直接复用其他业务接口
- 优惠券需要接入确认订单、提交订单、取消订单三个核心链路

## 3. 现状核对

当前代码里已经存在优惠券相关占位，但未真正实现：

- `OrderParam.couponIds`
- `ShopCartOrderDto.couponReduce`
- `ShopCartOrderDto.coupons`
- `ShopCartOrderMergerDto.coupons`
- `CouponOrderDto` 为空对象

订单主链路当前通过事件监听完成：

- 确认订单：`ConfirmOrderEvent` -> `ConfirmOrderListener`
- 提交订单：`SubmitOrderEvent` -> `SubmitOrderListener`
- 取消订单：`MyOrderController.cancel` -> `OrderService.cancelOrders`

因此优惠券实现必须顺着上述链路接入，而不是绕开订单事件直接在 Controller 中硬编码。

## 4. 实施原则

- 先完成数据层和独立优惠券模块，再接订单链路
- 每一步都必须可单独编译验证
- 优惠券状态变更必须用条件更新，依据受影响行数判断成功
- 除第 1 步外，不提前写前端页面；优先保证后端闭环
- 每一步完成后先做规格审查，再做代码质量审查，通过后才进入下一步

## 5. 分步任务

### 任务 1：建立优惠券基础数据层与后台 CRUD

目标：

- 新增 `tz_coupon` 表结构脚本
- 新增优惠券实体、Mapper、Service、后台 Controller
- 实现后台分页、详情、创建、作废、按用户发券
- 补充必要的入参/出参对象与查询条件

涉及范围：

- `db/yami_shop.sql`
- `yami-shop-bean`
- `yami-shop-service`
- `yami-shop-admin`

约束：

- 字段、索引、状态定义必须与设计文档一致
- 后台接口保持项目现有风格
- 发券接口只做“发给指定用户”
- 此步不接入订单确认/提交/取消

完成标准：

- 后台存在独立优惠券管理接口
- 可以创建公开领取券与指定用户券
- 可以分页查询、查看详情、作废
- 可以直接给指定用户发放一张券

建议验证：

- `mvn -pl yami-shop-admin -am -DskipTests compile`

### 任务 2：实现用户领券与我的优惠券列表

目标：

- 新增用户侧领券接口
- 新增“我的优惠券”列表接口
- 实现公开券从待发放到未使用的领取逻辑

涉及范围：

- `yami-shop-api`
- `yami-shop-service`
- `yami-shop-bean`

约束：

- 仅允许领取 `status=0` 且处于有效期内的公开券
- 领取动作必须用条件更新保证并发安全
- 领取后绑定 `user_id`、写入 `receive_time`

完成标准：

- 用户可通过 `couponNo` 领取一张公开券
- 用户可查询自己名下优惠券列表

建议验证：

- `mvn -pl yami-shop-api -am -DskipTests compile`

### 任务 3：接入确认订单优惠券试算展示

目标：

- 在确认订单阶段返回可用券/不可用券
- 计算选中优惠券对单店订单金额的试算结果
- 补全 `CouponOrderDto` 结构

涉及范围：

- `yami-shop-api`
- `yami-shop-service`
- `yami-shop-bean`

约束：

- 本步只做展示和试算，不改券状态
- 一单只允许一张券；若前端传多个 `couponIds`，后端按最小实现只接受一张，否则报错
- 金额判定基于服务端重新计算结果

完成标准：

- `/p/order/confirm` 返回可用于当前订单的优惠券信息
- 选券后能在确认单金额中体现 `couponReduce`

建议验证：

- `mvn -pl yami-shop-api -am -DskipTests compile`

### 任务 4：接入提交订单核销

目标：

- 在提交订单时重新校验券归属、状态、时间、门槛
- 下单成功后将优惠券从未使用改为已使用
- 记录 `order_number`、`use_time`

涉及范围：

- `yami-shop-api`
- `yami-shop-service`
- `yami-shop-bean`

约束：

- 核销必须在订单事务内完成
- 条件更新必须带 `status=1`
- 若核销失败，订单提交整体失败回滚

完成标准：

- 使用优惠券提交订单成功后，券状态变为已使用
- 重复提交或并发核销时只能成功一次

建议验证：

- `mvn -pl yami-shop-api -am -DskipTests test`
- 至少补一组服务层或监听器相关测试

### 任务 5：接入取消订单退券

目标：

- 订单取消时，如果使用了优惠券，则回退为未使用
- 清空 `order_number`、`use_time`

涉及范围：

- `yami-shop-service`
- `yami-shop-api`

约束：

- 只允许回退属于当前订单号的已使用券
- 退券逻辑必须幂等，避免重复取消带来脏数据

完成标准：

- 取消未支付订单后，已使用的优惠券自动退回未使用

建议验证：

- `mvn -pl yami-shop-api -am -DskipTests test`

### 任务 6：补充联调与测试文档

目标：

- 新增优惠券后端联调说明
- 补充管理员发券、用户领券、确认订单、提交订单、取消订单验证路径
- 说明调试 Token 方案在优惠券验证中的使用方式

涉及范围：

- `doc/`
- `docs/`

完成标准：

- 测试同学可按文档完成最小闭环验证

## 6. 执行顺序

严格按以下顺序推进，不并行改动：

1. 任务 1：建立优惠券基础数据层与后台 CRUD
2. 任务 2：实现用户领券与我的优惠券列表
3. 任务 3：接入确认订单优惠券试算展示
4. 任务 4：接入提交订单核销
5. 任务 5：接入取消订单退券
6. 任务 6：补充联调与测试文档

## 7. 风险提醒

- 当前订单参数是 `couponIds` 列表，但本次只支持单券，接入时必须明确约束
- 订单取消链路目前只处理库存回滚，新增退券逻辑时要避免破坏现有库存回补流程
- 若修改公共模块依赖，运行 `spring-boot:run` 前需要先执行 `mvn install`，否则可能仍加载旧依赖产物

## 8. 当前执行策略

本次采用“主会话把控 + 子智能体串行执行”的方式：

- 主会话负责拆任务、审查结果、决定是否进入下一步
- 子智能体每次只执行一个任务
- 当前首先执行 `任务 1：建立优惠券基础数据层与后台 CRUD`
