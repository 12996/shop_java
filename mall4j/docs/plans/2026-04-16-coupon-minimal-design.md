# 优惠券最小功能设计

## 1. 目标

本次优惠券功能只做最小可落地版本，目标是尽快打通“创建券 -> 发券/领券 -> 下单使用 -> 取消退券”的完整链路。

本次范围只包含：

- 固定金额满减券
- 一单只允许使用一张券
- 后台直接发券给指定用户
- 用户主动领取优惠券
- 订单提交时核销
- 订单取消时退回

本次明确不做：

- 折扣券
- 多券叠加
- 券模板体系
- 复杂营销活动
- 最优优惠组合
- 转赠、分享、裂变

## 2. 总体方案

采用单表方案，新增 `tz_coupon` 表。

设计原则：

- 一行数据就是一张真实优惠券
- 每张券有一个固定唯一标识 `coupon_no`
- 不区分模板和实例
- 用最少的数据结构支撑后台发券和用户领券两种方式

这样做的原因是：

- 开发速度最快
- 状态最直观
- 查询和排障成本低
- 适合当前“先做能用版本”的目标

## 3. 数据表设计

表名：`tz_coupon`

建议字段：

- `coupon_id`：主键
- `coupon_no`：优惠券唯一标识，唯一索引
- `coupon_name`：优惠券名称
- `reduce_amount`：优惠金额
- `condition_amount`：使用门槛，满多少可用
- `status`：状态，`0待发放`、`1未使用`、`2已使用`、`3已过期`
- `source_type`：来源，`1后台发放`、`2用户领取`
- `receive_mode`：领取方式，`1后台指定用户发放`、`2前台公开领取`
- `user_id`：所属用户，未领取时为空
- `order_number`：使用该券的订单号
- `start_time`：生效时间
- `end_time`：失效时间
- `receive_time`：发放/领取时间
- `use_time`：使用时间
- `remark`：后台备注
- `create_time`
- `update_time`

建议索引：

- `uk_coupon_no(coupon_no)`
- `idx_user_status(user_id, status)`
- `idx_end_time(status, end_time)`

## 4. 状态流转

优惠券状态固定如下：

- `0`：待发放
- `1`：未使用
- `2`：已使用
- `3`：已过期

状态流转规则：

- 创建公开领取券时，初始为 `0待发放`
- 后台直接发券给用户时，直接变为 `1未使用`
- 用户主动领取成功时，从 `0待发放` 变为 `1未使用`
- 用户提交订单并核销成功后，从 `1未使用` 变为 `2已使用`
- 订单取消后，如已使用该券，则从 `2已使用` 回退到 `1未使用`
- 超过 `end_time` 后，变为 `3已过期`

约束规则：

- `0待发放` 的券不允许下单使用
- `1未使用` 的券必须已经绑定 `user_id`
- `2已使用` 的券必须记录 `order_number`
- `3已过期` 的券不能再领取和使用

## 5. 接口设计

### 5.1 后台接口

- `POST /admin/coupon`
  - 创建优惠券

- `POST /admin/coupon/send`
  - 发券给指定用户

- `GET /admin/coupon/page`
  - 分页查询优惠券

- `GET /admin/coupon/{id}`
  - 查看优惠券详情

- `POST /admin/coupon/{id}/invalidate`
  - 作废优惠券

### 5.2 用户侧接口

- `GET /p/myCoupons`
  - 查询我的优惠券

- `POST /p/coupon/receive/{couponNo}`
  - 按券码领取优惠券

- `GET /p/order/confirm`
  - 查询当前订单可用/不可用优惠券

- `POST /p/order/submit`
  - 提交订单，携带 `couponId`

## 6. 下单接入设计

### 6.1 确认订单

`confirmOrder` 阶段只负责计算和展示：

- 查询当前用户名下 `status=1` 的优惠券
- 过滤掉未生效、已过期、不满足金额门槛的优惠券
- 返回可用券和不可用券列表给前端

注意：

- 该阶段不能修改优惠券状态
- 该阶段只能做展示和试算

### 6.2 提交订单

`submitOrder` 阶段才做最终校验和核销：

- 校验券是否存在
- 校验券是否属于当前用户
- 校验券状态是否为 `未使用`
- 校验是否在有效期内
- 校验订单金额是否满足门槛
- 校验减免金额是否合法

校验通过后：

- 参与订单金额计算
- 订单创建成功后，把优惠券更新为 `已使用`
- 记录 `order_number` 和 `use_time`

### 6.3 取消订单

如果订单使用了优惠券，则取消订单时：

- 把券状态从 `已使用` 回退为 `未使用`
- 清空 `order_number`
- 清空 `use_time`

## 7. 金额计算规则

本次固定为：

- 商品金额
- 减去店铺活动优惠
- 减去优惠券优惠
- 加上运费
- 得到最终应付金额

约束：

- 一单只允许一张券
- 优惠券不能把订单金额减成负数
- 所有金额以服务端重新计算结果为准

## 8. 并发与数据一致性

最容易出问题的场景：

- 同一张公开券被重复领取
- 同一张券被重复下单核销
- 订单取消时退券遗漏

处理原则：

- 领取券时必须做条件更新
- 核销券时必须做条件更新
- 更新条件必须带上当前 `status`
- 是否成功以更新行数为准，而不是以前端传参为准

建议做法：

- 用户领取时：`status=0` 才允许改为 `1`
- 下单核销时：`status=1` 才允许改为 `2`
- 取消退券时：必须校验 `order_number` 与当前订单一致

## 10. 代码复用策略

本功能可以复用当前项目后台模块的实现结构和代码模式，但不直接复用现有业务接口。

明确原则：

- 复用 `Controller + Service + Mapper + Model + DTO/VO` 的实现结构
- 复用后台分页、详情、保存、更新、删除的开发模式
- 复用 MyBatis Plus / Mapper XML 的实现方式
- 复用后台管理端列表页和表单页的交互风格

明确不做：

- 不直接复用商品、分类、公告等已有业务接口
- 不把优惠券业务动作混入现有无关模块接口

原因：

- 优惠券有独立业务字段和状态流转
- 发券、领券、作废、核销、退券都是优惠券特有行为
- 单独建一组优惠券接口更清晰，也更容易后续扩展

结论：

- 建表需要新建
- 后台 CRUD 需要新建优惠券模块接口
- 但实现结构按现有后台模块模式复用

## 11. 推荐实施顺序

1. 建表和后台 CRUD
2. 后台发券
3. 用户领取优惠券
4. 我的优惠券列表
5. 确认订单展示优惠券
6. 提交订单核销优惠券
7. 取消订单退券

## 12. 当前结论

本次优惠券最小方案采用单表方案，功能简单、边界明确，适合先快速落地验证。

后续如果需要扩展：

- 券模板
- 多券叠加
- 限领规则
- 新人券/活动券

再考虑从单表实例券模型升级为“模板表 + 用户券表”。
